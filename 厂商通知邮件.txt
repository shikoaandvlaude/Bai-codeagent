JeecgBoot 团队，你们好：

我是一名安全研究人员，近期在分析 JeecgBoot 源码时发现一个 SQL 注入漏洞，
现将详细情况报告如下，希望能协助你们尽快修复。

----------------------------------------------------------------------
一、问题概述
----------------------------------------------------------------------

今年3月31日，你们通过 issue #9434 修复了 QueryGenerator 类中的 SQL 注入
问题，在 installMplus() 和 installAuthMplus() 中增加了
SqlInjectionUtil.filterContent() 过滤。

但同类中的第三个方法 installAuthJdbc()（第949-952行）被遗漏了，没有加
过滤。该方法通过 getSqlRuleValue() 获取规则值后直接拼接到 StringBuffer，
最终经 MyBatis XML 的 ${permissionSql} 字符串替换执行，存在 SQL 注入风险。

----------------------------------------------------------------------
二、漏洞位置
----------------------------------------------------------------------

文件: QueryGenerator.java
方法: installAuthJdbc()
行号: 第951行

漏洞代码:
  sb.append(sqlAnd + getSqlRuleValue(ruleMap.get(c).getRuleValue()));

对比已修补的 installAuthMplus() 第996-1000行:
  String sqlRule = getSqlRuleValue(ruleMap.get(c).getRuleValue());
  SqlInjectionUtil.filterContent(sqlRule, null);
  queryWrapper.and(i -> i.apply(sqlRule));

----------------------------------------------------------------------
三、注入落地点
----------------------------------------------------------------------

文件: JeecgDemoMapper.xml 第12行
内容: select * from demo where 1=1 ${permissionSql}

使用了 ${} 字符串替换而非 #{} 预编译参数。

----------------------------------------------------------------------
四、利用条件
----------------------------------------------------------------------

攻击者需要具有管理员权限，能够在"数据权限规则"中配置 SQL_RULES_COLUMN
类型的规则。通过在 ruleValue 中写入恶意 SQL 片段（如 UNION SELECT），
可以在后续查询中跨表读取敏感数据。

我已在本地 H2 环境中验证，UNION 注入可成功读取其他表数据。

----------------------------------------------------------------------
五、建议修复方式
----------------------------------------------------------------------

与 #9434 相同的修复模式，第951行改为:

  String sqlRule = getSqlRuleValue(ruleMap.get(c).getRuleValue());
  SqlInjectionUtil.filterContent(sqlRule, null);
  sb.append(sqlAnd + sqlRule);

另外建议:
  - JeecgDemoMapper.xml 第10行的"暂不处理sql注入问题"注释应删除
  - 长期考虑将 ${permissionSql} 改为预编译方式
